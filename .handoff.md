## 2. COMPLETED (this session — 1 PR merged)

User said "continue" → picked up the explicitly-sequenced next step (PR-2) and shipped it:

- **#294 — per-settlement goods stocks (the per-settlement-stocks STORAGE SWAP, PR-2).**
  Moved the intermediate-goods ledger off the single nation-wide `intermediateGoodStocks`
  pool onto **per-settlement `Settlement.goodStocks?: Record<string,number>`** (optional,
  sparse). The #292 accessor seam made it mechanical — production logic untouched, only the
  backing store moved. **Aggregate-equivalent / byte-identical gameplay:**
  - `goodStock(id)` / `hasGoodStock(id)` — **sum / any** across towns (was one record read).
  - `produceGood(id,qty)` — **distributes** the tick's output across towns by the producing
    sector's output (`goodProducingSector`: agriculture for food/textiles, industry for the
    rest). Last contributing town gets the **exact remainder** → no float drift; Σ == qty.
    All sector outputs 0 (early game / bare fixture) → banks in the **capital**.
  - `drawGood(id,qty)` — **greedy in-order** subtraction across towns; aggregate drops by
    exactly `min(qty,total)` (matches old `max(0,pool−qty)`); only subtraction, so the floor
    lands on an exact value. Still a **no-op for an untracked raw**.
  - `seedGoodStock(id)` — seeds a 0-entry into the **capital** if untracked anywhere; never
    overwrites.
  - `goodStocksSnapshot()` — now a **derived aggregate** (sum across towns), NOT the
    serialize source. `restoreGoodStocks(pool)` — now **legacy-migration only**: deposits an
    old save's top-level pool into the capital, but only when no town already holds per-town
    stocks (so a new save isn't clobbered); no-op on undefined/empty.
  - **serialize:** per-town `goodStocks` rides each settlement's `...s` spread (sparse); the
    top-level `intermediateGoodStocks` key is **gone**. **deserialize:**
    `r.restoreGoodStocks(d.intermediateGoodStocks)` stays as the migration call.
  - **Why it was safe (KEY INSIGHT):** the stock ledger is **gameplay-inert in healthy
    play** — the solver's `rawLevel` callback only fires for **leaf raws** (coal/iron/…),
    which are never tracked in the ledger, so it reads the **sector proxy**, never a tracked
    good's magnitude; and the Supply UI panel reads the **live solve** (`supplyChainSnapshot`),
    not stock magnitudes. So the per-town split changes no observable behaviour today.

Tests 971 → **975** (+4: snapshot-aggregation, legacy-migration-into-capital, per-settlement
round-trip, new-save-format assertion). Determinism harness ✅, save-size guard ✅.

- **`goodStocksSnapshot()` is a DERIVED aggregate now**, not the serialized form. Each
  settlement serializes its own `goodStocks` via the `...s` spread. Don't re-add a top-level
  field.
- **`restoreGoodStocks` is migration-ONLY** — it deposits a legacy pool into the capital
  *only if no town already has per-town stocks*. Calling it on a populated sim is a no-op
  by design (guards against clobbering a freshly-spread new save in deserialize).
- **`drawGood` greedy order vs. `produceGood` weighted split** — draws drain towns in
  settlement-array order (exact, no division); produces split by sector output (division,
  remainder to the last town). Both preserve the aggregate; the *per-town* distribution is
  gameplay-inert today, so don't over-index on which town holds what until the
  consume/ship-between-towns PR makes it matter.
- **`drawGood` no-ops on an untracked good on purpose** — that's how a raw input
  (coal/iron/…) stays proxied by `sectorRawLevel`. Preserved through the swap.
